Two high-level contrasts are the most important:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{random\_matched} provides coverage without behavior quality;
    \item \texttt{dqn\_eps010} provides behavior quality without broad support.
\end{itemize}
These opposing properties explain why the same offline dataset can favor different learning paradigms.

\section{File Naming Conventions}
\label{app:naming}

To keep the code and data pipeline auditable, we adopt a simple file naming scheme.

\paragraph{Episode datasets.}
Raw collected trajectories are stored as
\begin{center}
\texttt{<dataset\_name>\_episodes.pkl}
\end{center}
with companion summaries
\begin{center}
\texttt{<dataset\_name>\_episodes.summary.json}.
\end{center}
Examples include:\begin{quote}\ttfamily\small cartpole\_random\_matched\_episodes.pkl\\ cartpole\_dqn\_eps010\_episodes.pkl\end{quote}

\paragraph{Transition datasets.}
Flattened replay-style data are stored as
\begin{center}
\texttt{<dataset\_name>\_transitions.pkl}
\end{center}
with companion summaries
\begin{center}
\texttt{<dataset\_name>\_transitions.summary.json}.
\end{center}
Examples include:\begin{quote}\ttfamily\small cartpole\_dqn\_eps010\_transitions.pkl\\ cartpole\_random\_matched\_transitions.pkl\\ cartpole\_mixed\_random20\_dqn010\_transitions.pkl\\ cartpole\_mixed\_random50\_dqn010\_transitions.pkl\end{quote}

\paragraph{Scripts.}
The codebase follows a stage-oriented naming style:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{generate\_*} for data collection;
    \item \texttt{flatten\_*} for episode-to-transition conversion;
    \item \texttt{train\_*} for model training;
    \item \texttt{evaluate\_*} for checkpoint re-evaluation; and
    \item \texttt{analyze\_*} for post-hoc log or dataset analysis.
\end{itemize}

This convention is useful not only for reproducibility, but also for the later zero-knowledge setting, where one wants a clean separation between data generation, dataset commitment, model training, and audited verification statements.

\section{Reporting Notes}
\label{app:reporting-notes}

We distinguish two different kinds of variability in the paper:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{within-seed evaluation variance}, reported as the standard deviation over the 100 evaluation episodes for a selected checkpoint; and
    \item \textbf{across-seed variance}, reported in aggregate rows as the standard deviation across seed-wise mean returns.
\end{itemize}

This distinction matters because a method can be internally variable within an evaluation rollout while still being consistent across random seeds, or vice versa. The strong seed sensitivity of offline Double DQN on \texttt{random\_matched} is an example of the second case.

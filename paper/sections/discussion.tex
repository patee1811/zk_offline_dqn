\subsection{What Has Been Checked}

The current system checks a precise stack of relation statements: a private
transition or minibatch belongs to a committed trajectory dataset; fixed-point
TD target, TD error, SmoothL1 loss, and public claimed outputs are recomputed
correctly; selected forward-TD rows anchor Q-values to committed MLP weights;
and one micro-scale SGD update is consistent with the checked gradients,
deltas, learning rate, and post-model commitment.

The main engineering result is that the canonical TD MVP statement is not only
checked by a Python script. It is implemented in a concrete SP1 zkVM backend
and validated with proof generation and verification on Kaggle. The wider
relation stack remains Python-oracle and report-backed unless separate SP1
validation summaries are produced for those fixtures.

\subsection{What Has Not Been Proved}

The paper deliberately avoids claiming full proof-of-training. The current SP1
backend still does not prove:
\begin{itemize}[leftmargin=1.5em]
    \item checkpoint-to-checkpoint consistency;
    \item target-network synchronization over long traces;
    \item replay-sampling correctness over a full training run;
    \item Adam optimizer state;
    \item model selection, early stopping, or best-checkpoint choice.
\end{itemize}
The witness-Q TD relation remains useful as a smaller layer, while the
forward-TD MLP and tiny one-step SGD relations cover selected model-grounded
and update-grounded cases.

\subsection{Why the Scoped Claim Is Still Useful}

A narrow claim is not a weakness if it is the right decomposition point.
Offline DQN training contains several distinct verifiability problems:
membership in committed trajectories, TD arithmetic, neural-network evaluation,
gradient computation, optimizer semantics, and trace composition. The present
work validates one backend layer and gives a tested Python artifact path for
future trace composition. In a Q1 systems/security framing, the contribution should be read
as a reproducible ZK-backed verification pipeline for selected RL-specific
relations, not as an end-to-end zkML training system.

\subsection{Scalability}

The TD MVP SP1 validation is prototype-scale. Proof generation takes minutes,
while verification is fast. The proof size is recorded for the canonical
vector. This is encouraging for artifact-level verification,
but it is far from sufficient for full training traces. Scaling to practical
training proofs will require aggregating many update steps and likely using
recursion or specialized circuits for repeated linear algebra.

\subsection{Benchmark Scope}

The TD-2/4/8 fixtures now use distinct committed transitions. The benchmark is
still small-scale: CartPole and MountainCar are classic-control tasks, and the
MountainCar dataset is a seeded random-policy artifact rather than a
policy-performance result.

\subsection{Submission Positioning}

For a Q1 journal submission, the paper should be positioned as a
security/systems artifact. The strongest defensible claim is:
\begin{quote}
relation-level offline DQN verification over committed trajectory data, with a
validated SP1 proof for the TD MVP canonical vector and Python semantic oracles
for model-grounded and update-grounded extensions.
\end{quote}
The paper should not be positioned as solving full verifiable RL training.
Reviewers should be able to see exactly which relation is proved and which
parts remain future work.

\subsection{What Has Been Proved}

The current system proves a precise statement: a private transition or minibatch
belongs to a committed trajectory dataset, and the fixed-point TD target, TD
error, SmoothL1 loss, and public claimed outputs are recomputed correctly. This
is a meaningful RL-specific statement because it includes terminal branch
semantics, Bellman backup arithmetic, committed replay membership, and
minibatch aggregation.

The main engineering result is that this statement is not only checked by a
Python script. It is implemented in a concrete SP1 zkVM backend, benchmarked
with proof generation and verification, and tested against a negative suite
that agrees with the Python semantic oracle.

\subsection{What Has Not Been Proved}

The paper deliberately avoids claiming full proof-of-training. The current SP1
backend does not prove:
\begin{itemize}[leftmargin=1.5em]
    \item neural-network forward passes;
    \item argmax action selection;
    \item target-network forward evaluation;
    \item SmoothL1 gradients or backpropagation;
    \item optimizer updates;
    \item checkpoint-to-checkpoint consistency;
    \item target-network synchronization over long traces;
    \item replay-sampling correctness over a full training run;
    \item model selection, early stopping, or best-checkpoint choice.
\end{itemize}
The Q-values in the SP1 TD statement are witness values. Python forward
consistency checks exist, but they are outside the ZK backend.

\subsection{Why the Scoped Claim Is Still Useful}

A narrow claim is not a weakness if it is the right decomposition point.
Offline DQN training contains several distinct verifiability problems:
membership in committed trajectories, TD arithmetic, neural-network evaluation,
gradient computation, optimizer semantics, and trace composition. The present
work resolves the first backend layer and gives a tested artifact path for the
next layers. In a Q1 systems/security framing, the contribution should be read
as a reproducible ZK-backed verification pipeline for the RL-specific TD layer,
not as an end-to-end zkML training system.

\subsection{Scalability}

The locked SP1 benchmark is prototype-scale. Proof generation takes minutes for
the TD-1/2/4/8 cases, while verification is fast. Proof size is roughly stable
across the tested cases. This is encouraging for artifact-level verification,
but it is far from sufficient for full training traces. Scaling to real update
proofs will require proving neural-network operations, aggregating many update
steps, and likely using recursion or specialized circuits for repeated linear
algebra.

\subsection{Benchmark Scope}

The TD-2/4/8 fixtures repeat the canonical TD item to exercise multiple
memberships, per-item relation checks, and batch aggregation. This is useful
for validating the minibatch relation, but it is not yet a diverse sampled
replay-batch benchmark. A stronger future artifact should generate TD-2/4/8
fixtures from distinct committed transitions while preserving the same public
statement.

\subsection{Submission Positioning}

For a Q1 journal submission, the paper should be positioned as a
security/systems artifact. The strongest defensible claim is:
\begin{quote}
ZK-backed offline DQN TD and minibatch-TD verification over committed
trajectory data, with Python pre-ZK extensions for one-step update and
short-trace verification.
\end{quote}
The paper should not be positioned as solving full verifiable RL training.
Reviewers should be able to see exactly which relation is proved, which
relations are checked only in Python, and which parts remain future work.

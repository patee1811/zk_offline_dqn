\subsection{Evaluation Principle}

Unless otherwise noted, all claims in this section are based on \emph{100-episode re-evaluation} of the best checkpoint selected during training. We use short 10-episode evaluations only for model selection, because they are cheaper but noticeably noisier. This distinction is important: several runs show optimistic early spikes under 10-episode evaluation that do not fully survive longer re-evaluation.

\subsection{Main Benchmark on the Behavior Dataset}

\Cref{tab:dqn010-main} reports the main comparison on the narrow behavior dataset \texttt{dqn\_eps010}. Three patterns are immediate.

First, naive offline Double DQN is consistently weak on this dataset. Across the three seeds, its 100-episode re-evaluation means are only 31.50, 54.25, and 94.68. The aggregate across-seed mean is 60.14, which is far below the other baselines.

Second, behavior cloning is strong on the same dataset. Its seed-wise re-evaluation means are 392.87, 461.88, and 398.19, with an across-seed mean of 417.65. This indicates that \texttt{dqn\_eps010} contains useful action supervision even though it is not favorable to naive bootstrapped value learning.

Third, CQL-lite largely closes the gap between imitation and offline value learning. Its seed-wise re-evaluation means are 416.14, 341.79, and 447.42, with an across-seed mean of 401.78. While CQL-lite does not dominate BC on every seed, it dramatically outperforms naive offline Double DQN and is competitive with the strongest imitation baseline.

\subsection{Why the Mixed Datasets Do Not Help}

A natural hypothesis is that adding random transitions to a narrow behavior dataset should improve coverage and therefore rescue offline Double DQN. In the current experiments, that hypothesis is not supported. Both \texttt{mixed\_random50\_dqn010} and \texttt{mixed\_random20\_dqn010} remain poor under the present trainer.

This negative result is important. It suggests that transition-level mixing is not automatically beneficial: low-quality and high-quality data may interact in a way that degrades the target distribution seen by the value learner. Put differently, dataset composition matters not only through marginal coverage statistics, but also through how the induced backup targets behave during offline training.

\subsection{Short-Trace Verification with Deterministic Sampling-Rule Enforcement}

We further strengthened the pre-ZK short-trace prototype by adding deterministic sampling-rule enforcement to the short verified training trace. Instead of accepting only explicitly provided minibatch indices, the exporter and verifier now both enforce a declared contiguous deterministic schedule with public batch size and start offset. Concretely, for step index $t$, batch size $k$, and start offset $s$, the expected minibatch is defined as:
\[
B_t = [s + tk,\; s + tk + 1,\; \dots,\; s + tk + (k-1)].
\]

This strengthens the statement from
``the provided minibatch is valid and the update trace is correct''
to
``the provided minibatch is valid, was chosen according to a declared public rule, and the update trace is correct.''

Our initial experiments established successful verification for 2-step traces with target synchronization applied every two steps. We then extended the benchmark to 4-step and 8-step traces, again with synchronization applied every two steps. Finally, we added deterministic sampling-rule enforcement to the 8-step benchmark using public start offsets and batch sizes. Across all tested runs, artifact export completed successfully, verification completed successfully, and the final result was accepted by the trace verifier.

The progression from 2-step to 4-step and then to 8-step traces shows that the current short-trace pipeline is not limited to a tiny hand-crafted composition. Instead, it already supports longer multi-step verification while preserving checkpoint chaining and target-network synchronization consistency across the trace. The deterministic-sampling extension further strengthens this result: the verifier now checks not only that each update step is locally correct and globally chained, but also that the per-step minibatch indices match a declared public contiguous schedule.

The measured timings also suggest a simple scaling pattern in the current Python prototype. When the number of steps increases from 2 to 4 and then to 8, both export time and verification time increase in an approximately controlled manner. By contrast, within the tested 8-step traces, increasing the per-step minibatch size from 4 to 8 causes only a modest change in verification time. In this current range, the cost appears more sensitive to trace length than to this small change in per-step batch size.

These results should still be interpreted as preliminary pre-ZK consistency results rather than cryptographic proving-time measurements. The current short-trace verifier checks only a small number of explicitly specified update steps, with a fixed SGD optimizer, a declared target synchronization schedule, and now a declared deterministic sampling rule. It does not yet verify long training traces beyond the present short-trace scale, general replay-sampling correctness, checkpoint selection logic, or a true zero-knowledge backend. Nevertheless, the short-trace results provide evidence that the project has already moved beyond isolated update verification toward verification of a small but nontrivial training process under a stricter public statement.
